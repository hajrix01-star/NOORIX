## تفاصيل التنفيذ حسب النقاط

### 1. قاعدة البيانات (PostgreSQL، لا localStorage)

- **القرار:** PostgreSQL كقاعدة البيانات الوحيدة؛ Prisma ORM للإدارة والهجرات.
- **إلغاء:** أي حفظ في المتصفح (localStorage) للبيانات المالية أو المرجعية.
- **الأداء:** تعريف Indexes على الحقول الحرجة (company_id، transaction_date، reference_type، reference_id، إلخ) لدعم استعلامات ضخمة (100k+ فاتورة).
- **الملفات:** إزالة منطق localStorage من الواجهة؛ ربط كل الحفظ بالـ API (NestJS/Backend).

### 2. الدقة المالية (Decimal.js حصرياً)

- **القرار:** عدم استخدام `Number` أو `Math.round` في أي حساب مالي في الواجهة أو السيرفر.
- **الواجهة:** إعادة كتابة `src/constants/tax.js` (دالة splitTax وغيرها) باستخدام Decimal.js فقط؛ مراجعة unifiedTransaction، SalesService، SalesInvoiceScreen وأي ملف يلمس مبالغ.
- **السيرفر:** تخزين المبالغ كنوع Decimal في Prisma؛ إرجاعها كـ string أو صيغة متوافقة مع Decimal في الـ API.

### 3. Foreign Key Constraints (منع حذف مورد مرتبط)

- **القرار:** في schema Prisma (PostgreSQL)، ربط جدول الفواتير/الحركات بجدول الموردين عبر FK.
- **السلوك:** محاولة حذف مورد مرتبط بفواتير إما تُرفض من قاعدة البيانات أو تُعالج في طبقة الخدمة (منع الحذف أو أرشفة فقط).

### 4. التوقيت (توقيت السعودية UTC+3)

- **القرار:** توليد entry_date وكل timestamps حركات من السيرفر باستخدام timezone Asia/Riyadh (مثلاً TZ=Asia/Riyadh أو مكتبة توقيت في NestJS).
- **الملفات:** في Backend عدم الاعتماد على `new Date().toISOString()` بدون تحويل إلى UTC+3؛ في الواجهة إبقاء transaction_date من المستخدم مع إرساله للخادم.

### 5. الهوية البصرية (margin-right: 260px)

- **القرار:** معيار ثابت 260px لمنع تداخل القائمة الجانبية مع المحتوى.
- **الملف:** `src/index.css` — استبدال القيمة الحالية (220px) بـ 260px في قواعد `.app-main` لـ RTL (ومقابلها لـ LTR إن لزم).

### 6. القيد المزدوج (TreasuryTransactions / Ledger)

- **القرار:** كل حركة "حفظ فاتورة" (أو صرف/دخل/تحويل) تُترجم آلياً إلى قيد مزدوج في جدول الحركات (ledger_entries أو ما يعادله): مدين/دائن، مع ربط الخزينة والحساب.
- **التنفيذ:** في Backend، داخل نفس الـ Transaction، كتابة سطرين (أو السطور المطلوبة) حسب نوع العملية؛ لا حفظ فاتورة بدون قيد مقابل.

### 7. عزل البيانات (company_id إلزامي)

- **القرار:** كل استعلام قراءة/كتابة يمر عبر فلتر company_id مستمد من الجلسة أو JWT (وليس من body الطلب للعمليات الحساسة).
- **التنفيذ:** Guard أو Middleware في NestJS يضمن وجود company_id صالح؛ رفض أي طلب لا يحمل سياق شركة. getCompanies (أو ما يعادلها) يعيد فقط الشركات المسموح للمستخدم بها.

### 8. Server-side Pagination

- **القرار:** جميع قوائم البيانات (فواتير، موردين، حركات، إلخ) تُجلب عبر APIs تدعم page و pageSize و total.
- **التنفيذ:** نقاط API في Backend ترجع قطعاً محددة؛ الواجهة تعرض الترقيم وتبديل الصفحات دون تحميل كل السجلات دفعة واحدة.

### 9. Prisma Transactions (الكل أو لا شيء)

- **القرار:** أي عملية مالية مركبة (مثلاً فاتورة + قيد مزدوج + تحديث كاش) تُنفَّذ داخل `prisma.$transaction(...)` بحيث عند فشل أي خطوة يتم rollback كامل.
- **التنفيذ:** استبدال/ربط runInTransaction في الواجهة باستدعاء endpoint في Backend ينفذ العملية داخل transaction واحدة.

### 10. Unique Constraints (أرقام الفواتير)

- **القرار:** في schema Prisma، تعريف unique مركب (مثلاً company_id + supplier_id + invoice_number) لجدول الفواتير (أو ما يعادله) لمنع تكرار نفس رقم الفاتورة لنفس المورد.
- **السلوك:** محاولة إدخال مكرر تُرفض من DB أو من طبقة التحقق مع رسالة خطأ واضحة.

### 11. React Query (الكاش وإدارة الحالة)

- **القرار:** اعتماد React Query (TanStack Query) لإدارة جلب البيانات والكاش وتقليل الطلبات المكررة عند التنقل.
- **التنفيذ:** استبدال أو دمج useDataManager الحالي مع React Query؛ تعريف query keys تتضمن company_id؛ إبطال الكاش عند العمليات الكتابية (مثل unifiedTransaction أو invalidation من الخادم).

### 12. SmartTable (مكوّن مركزي + RTL)

- **القرار:** بناء مكوّن SmartTable مركزي واحد يستخدم في كل الصفحات التي تعرض جداول.
- **المتطلبات:** دعم RTL تلقائياً (بدون تعديل يدوي في كل صفحة)، checkbox للصفوف، عرض Total و Tax في التذييل، وتوحيد التنسيق (حسب .cursorrules).

### 13. Global Error Boundary

- **القرار:** وضع Error Boundary على مستوى التطبيق (مثلاً في App.jsx) يلتقط الأخطاء في أي مكوّن فرعي ويمنع انهيار الصفحة بالكامل.
- **التنفيذ:** مكوّن React يستخدم componentDidCatch (أو معادل في دالة) ويعرض واجهة استبدالية (رسالة خطأ، زر إعادة تحميل أو العودة).

### 14. Indexing و Full-text search (PostgreSQL)

- **القرار:** استخدام indexes على الأعمدة المستخدمة في الفلترة والترتيب (company_id، التواريخ، reference_type، إلخ).
- **البحث:** تفعيل إمكانية Full-text search في PostgreSQL للحقول النصية المطلوبة (مثلاً أسماء الموردين، أرقام الفواتير، أوصاف) للوصول السريع.

### 15. Audit Log (القيم القديمة والجديدة)

- **القرار:** جدول مستقل (مثلاً audit_log) يسجل العمليات الحساسة (تعديل مالي، تغيير صلاحيات، إلخ) مع القيمة القديمة والجديدة ومن قام ومتى.
- **التنفيذ:** في Backend، بعد أي تحديث/حذف منطقي لسجل مالي (أو حسب السياسة)، إدراج سطر في audit_log (داخل نفس الـ transaction إن أمكن).

### 16. Zod/Joi (التحقق على السيرفر)

- **القرار:** استخدام Zod أو Joi (أو class-validator في NestJS) للتحقق من صحة البيانات عند دخولها الـ API قبل الكتابة إلى القاعدة.
- **التنفيذ:** DTOs أو schemas لكل نقطة دخول حرجة؛ رفض الطلبات غير الصحيحة مع رسائل خطأ موحدة (وربما ربط بأكواد docs/ERROR_CODES.md).

### 17. Session Timeout (الجلسات الخاملة)

- **القرار:** آلية لإغلاق الجلسة تلقائياً بعد مدة خمول محددة (و/أو انتهاء صلاحية JWT).
- **التنفيذ:** على السيرفر: JWT بانتهاء صلاحية معقول؛ اختيارياً refresh token. على الواجهة: كشف خمول (مثلاً عدم نشاط) وتنبيه المستخدم أو تسجيل خروج تلقائي بعد المهلة.

## The Great Transition — ترتيب الأولوية الصارم (4 مراحل)

### المرحلة الأولى: بناء النخاع الشوكي (Backend & DB)

- إنشاء مشروع **NestJS** فوراً.
- تحويل الـ Schema الورقية إلى ملف **`schema.prisma`** حقيقي يطبق:
  - **FK** (Foreign Key) على كل العلاقات (مورد ← فواتير، شركة ← حركات، إلخ).
  - **Unique Constraints** (مثلاً company_id + supplier_id + invoice_number).
  - **Decimal** لكل الحقول المالية.
  - **Indexes** على company_id، transaction_date، reference_type، reference_id لدعم 100k+ فاتورة.
- **تحويل النقاط:** (1) قاعدة البيانات + 100k، (3) FK منع حذف مورد، (7) عزل company_id في Schema، (10) Unique أرقام فواتير، (14) Indexing و Full-text — → **نعم**.

### المرحلة الثانية: محرك الحسابات والزمن (Math & Time)

- استبدال **Math.round** بـ **Decimal.js** في ملف **`mathEngine.js`** مركزي (وإعادة كتابة `tax.js` / splitTax عليه).
- ضبط **moment-timezone** (أو بديل) لفرض توقيت **Asia/Riyadh** في السيرفر عند توليد entry_date وكل timestamps الحركات.
- **تحويل النقاط:** (2) الدقة المالية Decimal.js حصرياً، (4) التوقيت UTC+3 — → **نعم**.

### المرحلة الثالثة: الأمان والتدقيق (Security & Audit)

- تفعيل **Prisma Transactions** (`$transaction`) لضمان القيد المزدوج: كل فاتورة/صرف/تحويل يُكتب فيها سطرا المدين والدائن (أو السطور المطلوبة) داخل نفس الـ transaction.
- إنشاء جدول **`AuditLogs`** وبرمجة تسجيل (في الكود أو عبر تريجر/دالة) **القيم القديمة والجديدة** لأي تعديل مالي أو عملية حساسة.
- تطبيق **Zod/Joi** (أو class-validator) على الـ API قبل الكتابة إلى القاعدة.
- **تحويل النقاط:** (6) القيد المزدوج، (9) ACID Transactions، (15) Audit Log، (16) التحقق على السيرفر — → **نعم**.

### المرحلة الرابعة: الواجهة الاحترافية (Pro UI)

- بناء مكوّن **`SmartTable`** الموحد بمسافة أمان **260px** ودعم **RTL** آلياً لكل الصفحات.
- تفعيل **Server-side Pagination** في جميع الجداول والـ APIs (page, pageSize, total).
- دمج **React Query** للكاش وإدارة الحالة، و **Error Boundaries** (Global) لحماية النظام من الانهيار.
- إضافة آلية **Session Timeout** (خمول + JWT expiry) لحماية الجلسات.
- **تحويل النقاط:** (5) 260px، (8) Pagination، (11) React Query، (12) SmartTable، (13) Error Boundary، (17) Session Timeout — → **نعم**.
